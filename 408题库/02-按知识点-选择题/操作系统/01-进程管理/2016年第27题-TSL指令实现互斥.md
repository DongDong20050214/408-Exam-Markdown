**题干**

使用TSL（Test and Set Lock）指令实现进程互斥，正确的是（）。A. 退出临界区的进程负责唤醒阻塞态进程  B. 等待进入临界区的进程不会主动放弃 CPU  C. 上述伪代码满足"让权等待"的同步准则  D. while(TSL(&lock))语句应在关中断状态下执行

**答案**：B

**考点分析**
- TSL是忙等待，不主动放弃CPU

---


## 解题思路

### 第一步：理解TSL指令
TSL（Test and Set Lock）：测试并设置锁
硬件原子操作，用于实现互斥
特点：忙等待（spin lock）

### 第二步：逐项分析
**A. 退出进程负责唤醒阻塞进程 ❌**
```
TSL是忙等待，进程不会阻塞
等待进程一直循环检测锁
没有阻塞态进程，无需唤醒
结论：错误
```

**B. 等待进程不会主动放弃CPU ✅**
```
TSL使用忙等待（busy waiting）
进程在while循环中不断测试锁
持续占用CPU，不主动放弃
这是TSL的特点
结论：正确
```

**C. 满足"让权等待"准则 ❌**
```
"让权等待"：等待时应释放CPU
TSL是忙等待，不释放CPU
违反"让权等待"原则
结论：错误
```

**D. 应在关中断状态执行 ❌**
```
TSL本身是硬件原子操作
不需要关中断保护
硬件保证原子性
结论：错误
```

### 第三步：确认答案
等待进程不会主动放弃CPU → 答案：**B**

## 易错点分析

### 易错点1：认为TSL需要关中断
❌ 错误：所有互斥操作都要关中断保护
✅ 正确：TSL是硬件原子指令，本身保证原子性

### 易错点2：混淆忙等待和阻塞等待
| 等待方式 | 占用CPU | 让权等待 | 实现方式 |
|---------|---------|---------|---------|
| 忙等待 | ✅ | ❌ | TSL、自旋锁 |
| 阻塞等待 | ❌ | ✅ | 信号量、管程 |

**TSL工作原理**：
```c
while(TSL(&lock)) {
    // 忙等待，不断测试
    // 不放弃CPU
}
// 进入临界区
临界区代码;
lock = 0;  // 释放锁
```

**优缺点**：
- 优点：简单，临界区短时适用
- 缺点：浪费CPU，违反让权等待
