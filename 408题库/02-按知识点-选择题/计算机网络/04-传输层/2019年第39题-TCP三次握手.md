# 2019年408真题第39题

## 题目

主机甲主动发起一个与主机乙的 TCP 连接，甲、乙选择的初始序列号分别为 2018 和 2046，第三次握手 TCP 段的确认序列号是（）。

A. 2018  
B. 2019  
C. 2046  
D. 2047

## 答案

D

## 考点分析

### 考点：TCP三次握手、序列号和确认号

本题考查TCP连接建立过程中三次握手的细节，特别是确认序列号的计算规则。

### 核心知识点：

**TCP三次握手过程**：
1. **第一次握手**（SYN）：客户端 → 服务器
2. **第二次握手**（SYN + ACK）：服务器 → 客户端
3. **第三次握手**（ACK）：客户端 → 服务器

**序列号规则**：
- **seq（序列号）**：本段数据的第一个字节的序号
- **ack（确认序列号）**：期望接收的下一个字节的序号
- **确认号 = 对方的序列号 + 1**（握手阶段）

### 三次握手详细过程：

**已知条件**：
- 主机甲（客户端）初始序列号：2018
- 主机乙（服务器）初始序列号：2046

**第一次握手（甲 → 乙）**：
- 甲发送：SYN = 1, seq = 2018
- 含义：甲请求建立连接，甲的初始序列号是 2018

**第二次握手（乙 → 甲）**：
- 乙发送：SYN = 1, ACK = 1, seq = 2046, ack = 2019
- 含义：
  - 乙同意连接，乙的初始序列号是 2046
  - 乙确认收到甲的SYN，期望甲的下一个序列号是 2019（2018 + 1）

**第三次握手（甲 → 乙）**：
- 甲发送：ACK = 1, seq = 2019, ack = **2047**
- 含义：
  - 甲确认收到乙的SYN
  - 甲期望乙的下一个序列号是 **2047**（2046 + 1）

**第三次握手的确认序列号 = 2047**

**答案：D（2047）**

### 图解三次握手：

```
主机甲（客户端）                     主机乙（服务器）
初始序列号：2018                     初始序列号：2046

第一次握手：
    SYN=1, seq=2018
    ------------------>

第二次握手：
                        SYN=1, ACK=1
                        seq=2046, ack=2019
    <------------------

第三次握手：
    ACK=1
    seq=2019, ack=2047
    ------------------>

连接建立完成
```

### 关键理解：

**为什么确认号要 +1？**
- TCP的确认号表示"期望收到的下一个字节"
- SYN和FIN标志位虽然不占用数据字节，但**逻辑上占用1个序列号**
- 因此确认SYN时，ack = seq + 1

**序列号的作用**：
- 标识数据字节的顺序
- 实现可靠传输（检测丢失、重复、乱序）
- 初始序列号应该随机选择（安全性考虑）

**确认号的作用**：
- 告诉对方"我已经收到了序列号小于ack的所有数据"
- 表示"我期望下一个收到的字节序号是ack"
- 累积确认机制

### 完整示例（带数据传输）：

```
握手完成后的序列号状态：
- 甲的下一个seq：2019
- 乙的下一个seq：2047

如果甲发送100字节数据：
- seq = 2019, 数据占用 2019~2118
- 乙回复：ack = 2119（期望下一个字节从2119开始）

如果乙发送200字节数据：
- seq = 2047, 数据占用 2047~2246
- 甲回复：ack = 2247（期望下一个字节从2247开始）
```

### 三次握手中各段的标志位：

| 握手次序 | 方向 | SYN | ACK | seq | ack |
|---------|------|-----|-----|-----|-----|
| 第一次 | 甲→乙 | 1 | 0 | 2018 | - |
| 第二次 | 乙→甲 | 1 | 1 | 2046 | 2019 |
| 第三次 | 甲→乙 | 0 | 1 | 2019 | **2047** |

### 记忆要点：

- **SYN占用1个序列号**（即使没有数据）
- 确认号 = 收到的seq + 占用的序列号数
- 第三次握手的ack = 服务器的初始seq + 1
- 握手完成后，双方进入ESTABLISHED状态

### 易错点：

- 选择2046（忘记+1）
- 混淆甲和乙的序列号
- 不理解SYN标志占用1个序列号
- 混淆seq和ack的含义

### 补充：为什么需要三次握手？

1. **防止旧连接请求**：避免已失效的连接请求到达服务器
2. **双向确认**：确认双方的发送和接收能力
3. **同步序列号**：双方交换并确认初始序列号

两次握手的问题：
- 无法确认客户端的接收能力
- 旧的连接请求可能导致资源浪费

四次握手：
- 没有必要，第二次握手已经合并了SYN和ACK

## 解题思路

### 第一步：理解题目核心
TCP三次握手的确认号规则：ack = 对方seq + 1（SYN占1个序号）

### 第二步：逐项分析
**A. 2018**
❌ 这是甲的初始序列号，不是确认号。第三次握手中甲的seq=2019（已经+1），ack应该确认乙的SYN段，与甲自己的初始序列号无关。

**B. 2019**
❌ 这是甲在第三次握手中的seq值（甲发送SYN后序列号从2018增长到2019），不是ack值。混淆了"甲当前的序列号"与"甲期望乙发送的下一个序列号"。

**C. 2046**
❌ 这是乙的初始序列号，忘记SYN标志位占用1个序列号。乙发送SYN(seq=2046)后，下一个序列号应为2047，因此甲的确认号应该是2047而非2046。

**D. 2047**
✅ 正确。第三次握手时，甲要确认收到乙的SYN段。乙的SYN段seq=2046，SYN占1个序号，因此甲期望乙下一个发送的序列号是2046+1=2047。TCP确认号含义：期望接收的下一个字节序号。

### 第三步：确认答案
D（2047）

## 易错点分析

### 易错点1：SYN占用序列号的理解
❌ 错误理解：认为第三次握手的ack应该等于服务器的初始序列号2046，忘记SYN标志位虽然不携带数据但占用1个序列号。或者混淆seq和ack的含义，误选2019(甲的序列号)。
✅ 正确理解：TCP确认号(ack)的含义是"期望接收的下一个字节序号"。第三次握手时，客户端甲要确认收到了服务器乙的SYN段。乙的SYN段序列号是2046，SYN标志位占用1个序列号，因此甲期望乙下一个发送的序列号是2046+1=2047。所以第三次握手的ack=2047。三次握手序列号规律：①甲发SYN(seq=2018) ②乙发SYN+ACK(seq=2046, ack=2019) ③甲发ACK(seq=2019, ack=2047)。答案D。记忆要点：ack=对方seq+1，SYN/FIN各占1个序列号。

### 易错点2：seq与ack的区别
| seq（序列号） | ack（确认号） | 关键区别 |
|-------|-------|------|
| 标识本方发送数据的第一个字节序号 | 标识期望对方发送的下一个字节序号 | seq是本方状态，ack是对对方的响应；第三次握手seq=2019(甲)，ack=2047(确认乙) |
