# 2016年第46题

**真题出处**：[2016年第46题](../../../01-真题原卷/2016年408真题.md#46)

**考点**：进程调度 | 优先级调度 | 动态优先级 | 饥饿问题

**题干**：
某进程调度程序采用基于优先数（priority）的调度策略，即选择优先数最小的进程运行，进程创建时由用户指定一个nice作为静态优先数。为了动态调整优先数，引入运行时间cpuTime和等待时间waitTime，初值均为0。进程处于执行态时，cpuTime定时加1，且waitTime置0；进程处于就绪态时，cpuTime置0，waitTime定时加1。请回答下列问题。

1) 若调度程序只将 nice 的值作为进程的优先数，即 priority = nice，则可能会出现饥饿现象，为什么?

2) 使用 nice、cpuTime 和 waitTime 设计一种动态优先数计算方法，以避免产生饥饿现象，并说明 waitTime 的作用。

**答案**：
46.【王道书2.2节】

1)【标准答案】

由于采用了静态优先数，当就绪队列中总有优先数较小的进程时，优先数较大的进程一直没有机会运行，因此会出现饥饿现象。（2分）

2)【标准答案】

优先数priority的计算公式为

$$
\text { priority } = \text { nice } + k 1 \times \text { cpuTime } - k 2 \times \text { waitTime }
$$

式中，k1 > 0, k2 > 0，分别用来调整 cpuTime 和 waitTime 在 priority 中所占的比例。(3 分) waitTime 可使长时间等待的进程优先数减小，从而避免出现饥饿现象。(1 分)

【分析】

如果一个进程的运行时间较长，它的 cpuTime 就会增加，从而会降低它的优先级；如果一个进程的等待时间较长，它的 waitTime 就会增加，从而会提高它的优先级。

【评分说明】

① 公式中包含 nice 给 1 分，利用 cpuTime 增大优先数给 1 分，利用 waitTime 减少优先数给 1 分；部分正确，酌情给分。

② 若考生给出包含 nice、cpuTime 和 waitTime 的其他合理的优先数计算方法，同样给分。

## 解题思路

### 第一步：分析饥饿产生原因
静态优先级priority=nice固定不变，若就绪队列总有nice小的进程，nice大的进程永远得不到调度，产生饥饿。

### 第二步：引入动态调整因子
- cpuTime：运行时间越长优先级越低（惩罚）
- waitTime：等待时间越长优先级越高（补偿）

### 第三步：构建公式
priority = nice + k1×cpuTime - k2×waitTime
- k1>0：增大cpuTime降低优先级，限制长时间运行
- k2>0：增大waitTime提高优先级，防止饥饿

## 易错点分析

### 易错点1：公式中waitTime符号用加号
❌ 错误公式：priority = nice + k1×cpuTime + k2×waitTime。这会让等待时间长的进程优先级变大（数值变大），而题目说"选择优先数最小的进程运行"，等待久的进程反而更难被调度，加剧饥饿。
✅ 正确理解：priority数值越小优先级越高。waitTime前必须是减号，等待时间越长，priority数值越小，优先级越高，才能避免饥饿。cpuTime前是加号，运行时间越长priority越大，优先级越低，鼓励短作业。

### 易错点2：cpuTime和waitTime的更新时机混淆
❌ 错误理解：认为进程在就绪态时cpuTime增加（因为在等待CPU），执行态时waitTime增加（因为其他进程在等它）。
✅ 正确理解：cpuTime记录实际占用CPU的时间，只在执行态增加；waitTime记录等待CPU的时间，只在就绪态增加。执行态时waitTime置0（不再等待），就绪态时cpuTime置0（没在运行）。这体现了动态优先级的核心：奖励等待者，惩罚占用者。
