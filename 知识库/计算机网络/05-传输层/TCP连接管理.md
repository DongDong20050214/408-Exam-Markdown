# TCP连接管理

## 常见提问

**Q: 为什么需要三次握手，两次不行吗？**
A: 
- 两次握手无法确认客户端的接收能力
- 旧的连接请求可能导致服务器资源浪费
- 三次握手确保双方的发送和接收能力都正常

**Q: SYN和FIN标志占用序列号吗？**
A: 占用。SYN和FIN各占用1个序列号，虽然不传输数据，但逻辑上占用。普通ACK不占用序列号。

**Q: 第三次握手时的ack是怎么算的？**
A: ack = 对方的seq + 1。例如服务器初始seq=2046，客户端第三次握手的ack=2047。

**Q: 为什么需要四次挥手，不能三次吗？**
A: TCP是全双工通信，两个方向需要独立关闭。服务器收到FIN后，可能还有数据要发送，所以ACK和FIN分开发送（第二次和第三次）。

**Q: TIME_WAIT状态的作用是什么？**
A: 
1. 确保最后的ACK能到达（如果丢失，对方会重传FIN）
2. 等待2MSL，确保旧连接的所有数据包都消失，不干扰新连接

**Q: 谁会进入TIME_WAIT状态？**
A: 主动关闭连接的一方（不是发起连接的一方）。客户端或服务器都可能进入。

**Q: 半关闭状态是什么？**
A: 一方关闭发送方向，但仍可接收数据。FIN_WAIT_2和CLOSE_WAIT就是半关闭状态。应用场景：客户端发完请求关闭发送，但仍需接收服务器的大量响应数据。

## 解题模板

**题型特征**：计算三次握手或四次挥手中的序列号/确认号

### 模板1：三次握手序列号计算
```
已知：客户端初始seq=x，服务器初始seq=y

第一次握手（客户→服务）：
- SYN=1, seq=x

第二次握手（服务→客户）：
- SYN=1, ACK=1
- seq=y, ack=x+1  （确认客户端的SYN）

第三次握手（客户→服务）：
- ACK=1
- seq=x+1, ack=y+1  （确认服务器的SYN）

握手后：
- 客户端下一个seq = x+1
- 服务器下一个seq = y+1
```

### 模板2：四次挥手序列号计算
```
假设：客户端主动关闭，当前seq=u，服务器当前seq=v

第一次挥手（客户→服务）：
- FIN=1, seq=u

第二次挥手（服务→客户）：
- ACK=1, seq=v, ack=u+1  （确认客户端的FIN）

[服务器可能继续发送数据]

第三次挥手（服务→客户）：
- FIN=1, ACK=1
- seq=w, ack=u+1  （w可能>v，如果发送了数据）

第四次挥手（客户→服务）：
- ACK=1
- seq=u+1, ack=w+1  （确认服务器的FIN）
```

**关键规则**：
- SYN/FIN占用1个序列号
- ack = 对方seq + 占用的序列号数
- 普通ACK不占用序列号

**常见变式**：
- 求第三次握手的ack → 服务器初始seq + 1
- 求握手后的序列号 → 初始seq + 1
- 判断是否占用序列号 → SYN/FIN占用，ACK不占

## 连接的建立（三次握手）

## 连接的建立（三次握手）

**目的**：
- 建立可靠的全双工通信连接
- 同步双方的初始序列号
- 确认双方的发送和接收能力

### 三次握手过程

```
客户端                                  服务器
CLOSED                                 LISTEN

第一次握手：SYN=1, seq=x
    |--------------------------------->|  SYN_RCVD
SYN_SENT
                                       
第二次握手：SYN=1, ACK=1, seq=y, ack=x+1
    |<---------------------------------|
                                       
第三次握手：ACK=1, seq=x+1, ack=y+1
    |--------------------------------->|
ESTABLISHED                            ESTABLISHED
```

**详细说明**：

1. **第一次握手（客户端→服务器）**
   - 客户端发送：SYN=1, seq=x
   - 含义：请求建立连接，初始序列号为x
   - 客户端进入SYN_SENT状态

2. **第二次握手（服务器→客户端）**
   - 服务器发送：SYN=1, ACK=1, seq=y, ack=x+1
   - 含义：同意连接，服务器初始序列号为y，确认收到客户端的SYN
   - 服务器进入SYN_RCVD状态

3. **第三次握手（客户端→服务器）**
   - 客户端发送：ACK=1, seq=x+1, ack=y+1
   - 含义：确认收到服务器的SYN
   - 双方进入ESTABLISHED状态

### 为什么需要三次握手？

**两次握手的问题**：
- 无法确认客户端的接收能力
- 旧的连接请求可能导致服务器资源浪费

**具体场景**：
```
1. 客户端发送SYN1（网络延迟，未到达）
2. 客户端超时重发SYN2
3. SYN2到达，建立连接，传输数据，关闭连接
4. 延迟的SYN1现在到达
5. 如果只有两次握手：
   - 服务器回复SYN+ACK
   - 服务器认为连接已建立，但客户端不理会
   - 服务器资源浪费
6. 三次握手：
   - 服务器回复SYN+ACK
   - 等待客户端第三次握手
   - 客户端不发送第三次握手（因为已关闭）
   - 服务器超时，释放资源
```

### 序列号规则

**SYN标志占用1个序列号**：
- 第一次握手：seq=x
- 第二次握手：ack=x+1（期望收到x+1）
- 第三次握手：seq=x+1, ack=y+1

**初始序列号选择**：
- 应该随机选择（安全性考虑）
- 防止序列号预测攻击

## 连接的释放（四次挥手）

**为什么需要四次？**
- TCP是全双工通信
- 双方都需要独立关闭自己的发送方向
- 一方关闭发送，仍可接收

### 四次挥手过程

```
客户端                                  服务器
ESTABLISHED                            ESTABLISHED

第一次挥手：FIN=1, seq=u
    |--------------------------------->|  CLOSE_WAIT
FIN_WAIT_1                             (半关闭状态)
                                       
第二次挥手：ACK=1, seq=v, ack=u+1
    |<---------------------------------|
FIN_WAIT_2                             
(半关闭状态)                            [服务器可能还有数据要发送]
                                       
第三次挥手：FIN=1, ACK=1, seq=w, ack=u+1
    |<---------------------------------|  LAST_ACK
TIME_WAIT                              
                                       
第四次挥手：ACK=1, seq=u+1, ack=w+1
    |--------------------------------->|  CLOSED
(等待2MSL)
CLOSED
```

**详细说明**：

1. **第一次挥手（客户端→服务器）**
   - 客户端发送：FIN=1, seq=u
   - 含义：客户端没有数据要发送了
   - 客户端进入FIN_WAIT_1状态

2. **第二次挥手（服务器→客户端）**
   - 服务器发送：ACK=1, ack=u+1
   - 含义：确认收到客户端的关闭请求
   - 服务器进入CLOSE_WAIT状态（半关闭）
   - 客户端进入FIN_WAIT_2状态

3. **第三次挥手（服务器→客户端）**
   - 服务器发送：FIN=1, ACK=1, ack=u+1
   - 含义：服务器也没有数据要发送了
   - 服务器进入LAST_ACK状态

4. **第四次挥手（客户端→服务器）**
   - 客户端发送：ACK=1, ack=w+1
   - 含义：确认收到服务器的关闭请求
   - 客户端进入TIME_WAIT状态
   - 服务器收到后进入CLOSED状态

### 为什么需要四次挥手？

**原因**：TCP是全双工通信

```
客户端关闭发送：FIN（第一次）
服务器确认：ACK（第二次）
  ↓
此时：客户端→服务器方向关闭
     服务器→客户端方向仍开放
  ↓
服务器可能还有数据要发送
  ↓
服务器发送完数据后关闭发送：FIN（第三次）
客户端确认：ACK（第四次）
```

### TIME_WAIT状态

**为什么需要TIME_WAIT？**

1. **确保最后的ACK能到达**
   - 如果最后的ACK丢失，服务器会重传FIN
   - 客户端需要在TIME_WAIT期间响应重传的FIN

2. **避免旧连接的数据干扰新连接**
   - 确保旧连接的所有数据包都消失
   - 防止旧数据包被新连接误接收

**TIME_WAIT持续时间**：
- 2MSL（Maximum Segment Lifetime）
- MSL：报文段最大生存时间（通常30秒到2分钟）
- 2MSL：确保来回的报文都消失

## TCP状态转换图

### 客户端状态转换

```
CLOSED → SYN_SENT → ESTABLISHED → FIN_WAIT_1 → FIN_WAIT_2 → TIME_WAIT → CLOSED
```

### 服务器状态转换

```
CLOSED → LISTEN → SYN_RCVD → ESTABLISHED → CLOSE_WAIT → LAST_ACK → CLOSED
```

### 完整状态转换表

| 状态 | 说明 | 触发条件 |
|------|------|---------|
| CLOSED | 关闭状态 | 初始/最终状态 |
| LISTEN | 监听状态 | 服务器等待连接 |
| SYN_SENT | 同步已发送 | 客户端发送SYN |
| SYN_RCVD | 同步已接收 | 服务器收到SYN |
| ESTABLISHED | 已建立连接 | 三次握手完成 |
| FIN_WAIT_1 | 终止等待1 | 主动关闭方发送FIN |
| FIN_WAIT_2 | 终止等待2 | 收到对方ACK |
| CLOSE_WAIT | 关闭等待 | 被动关闭方收到FIN |
| LAST_ACK | 最后确认 | 被动关闭方发送FIN |
| TIME_WAIT | 时间等待 | 主动关闭方收到FIN |
| CLOSING | 正在关闭 | 双方同时关闭（罕见） |

## 半关闭状态

**半关闭（Half-Close）**：
- 一方关闭发送方向，但仍可接收
- FIN_WAIT_2和CLOSE_WAIT就是半关闭状态

**应用场景**：
```
客户端：发送完请求后关闭发送
服务器：处理请求，发送大量响应数据
客户端：在半关闭状态接收服务器的数据
```

## 同时打开与同时关闭

### 同时打开（罕见）

双方同时发起连接：
```
客户端发送SYN → ← 服务器发送SYN
客户端收到SYN，发送SYN+ACK → ← 服务器收到SYN，发送SYN+ACK
```
最终建立连接

### 同时关闭（罕见）

双方同时发送FIN：
```
客户端FIN → ← 服务器FIN
客户端进入CLOSING状态
服务器进入CLOSING状态
互相确认后进入TIME_WAIT
```

## 典型例题

**例1**：为什么建立连接是三次握手，而关闭连接是四次挥手？

答：
- 建立连接时，服务器可以把SYN和ACK合并在第二次握手中发送
- 关闭连接时，服务器收到FIN后可能还有数据要发送
- 因此ACK和FIN分开发送（第二次和第三次）

**例2**：TIME_WAIT状态的作用？

答：
1. 确保最后的ACK能到达（如果丢失，可重发）
2. 等待2MSL，确保旧连接的数据包都消失

**例3**：客户端初始序列号2018，服务器初始序列号2046，第三次握手的ack是多少？

答：2047（服务器seq+1）

## 易错点

1. **SYN占用序列号**
   - SYN标志占用1个序列号
   - FIN标志也占用1个序列号
   - 普通ACK不占用序列号

2. **主动关闭方进入TIME_WAIT**
   - 不是发起连接方
   - 是主动关闭连接的一方

3. **四次挥手不能合并为三次**
   - 第二次和第三次之间可能有数据传输
   - 只有当服务器没有数据要发送时，才能合并

4. **半关闭状态理解**
   - 一个方向关闭，另一个方向仍开放
   - TCP是全双工，两个方向独立

## 记忆要点

**三次握手**：
- SYN → SYN+ACK → ACK
- SYN占用1个序列号
- 防止旧连接请求，确认双向能力

**四次挥手**：
- FIN → ACK → FIN → ACK
- FIN占用1个序列号
- 全双工需要独立关闭两个方向

**TIME_WAIT**：
- 主动关闭方
- 等待2MSL
- 确保可靠关闭

**状态记忆**：
- 主动打开：CLOSED → SYN_SENT → ESTABLISHED
- 被动打开：CLOSED → LISTEN → SYN_RCVD → ESTABLISHED
- 主动关闭：ESTABLISHED → FIN_WAIT_1 → FIN_WAIT_2 → TIME_WAIT → CLOSED
- 被动关闭：ESTABLISHED → CLOSE_WAIT → LAST_ACK → CLOSED
